# 07e — v1.1 Upgrade Report

> CRM consumes the full master Postgres schema. The bench universe expanded from 48 active cadences to the full 13,715 qualified ICP-fit prospects out of 190,692 total companies.

## Drizzle schema (introspection results)

**Total LOC of schema:** 532 lines (`src/db/schema.ts`).

Drizzle relations declared:
- `companies → many contacts, carriers, triggers, incumbent_peo`
- `companies → one cadence, one meddpicc, many notes, many tasks`
- `cadences → one company, many touches, one buyerCast, one meddpicc`
- `meddpicc → one company`
- `contacts/carriers/triggers/incumbentPeo → one company`

`AccountSummary` / `Account` type was widened to a permissive union of master + cadence fields so every existing component continues to compile against either the master projection or the legacy cadence shape.

## Server actions

`src/app/actions/index.ts` was reworked around a new `resolveCadence(idOrKey)` helper that resolves a row from either `cadences.company_id` (preferred) or `cadences.company_key` (legacy). All write paths (`logOutcome`, `addNote`, `addTask`, `toggleTask`, `disqualifyAccount`, `promoteToActive`, `updateMeddpicc`, `moveMeddpiccStage`) accept both id and key formats. Concretely:

- `addNote` / `addTask` — write to master `notes` / `tasks` keyed by `company_id`.
- `toggleTask` — now flips `tasks.status` between `"done"` and `"open"` (the master schema doesn't have a boolean `done` column).
- `disqualifyAccount` — flips `companies.disqualified=true` + `companies.disqualified_reason`, AND marks any active cadence as `DISQUALIFIED`.
- `promoteToActive` — inserts a new `cadences` row keyed by `company_id` if one doesn't already exist; otherwise updates the existing row. Touches are now FK'd via `cadence_id` (the new touches table dropped the `company_key` column).
- `logOutcome` — writes to `outcomes_ledger` with both `cadence_id` and `company_key`, plus the right `trigger_type` column name (the DB renamed `trigger` → `trigger_type`).

## v1.2 carry-overs

- **Tremor BarChart + DonutChart on /dashboard** — still in use. To remove the remaining Tremor footprint, ship hand-rolled SVG `<BarChart>` and `<DonutChart>` components matching the existing `<AreaChart>` pattern.
- **`triggers.still_active`** — the migration script never populates this column (it doesn't exist in the DB). v1.1 treats every trigger as currently active for score computation. v1.2 should backfill the column (or compute "active" from `trigger_date >= now() - interval '90 days'`) so the score reflects expiry correctly.
- **`incumbent_peo.peo_canonical`** — not in the DB; we display `peo_brand` as-is. v1.2 should add a canonicalization mapping (Insperity / TriNet / Paychex / Justworks / Questco / Other) and a stored `peo_canonical` column.
- **`outcomes_ledger.trigger`** — the legacy column name was `trigger`; the new DB uses `trigger_type`. `conversionByTrigger()` already targets `trigger_type`. The Python writer (`scripts/sync_to_neon.py`) needs to be updated to use the new column name before any new outcomes can be logged via the legacy writer.
- **Schedule A column for HEALTH-only carriers** — `carriers.benefit_type` is `NULL` for the bulk-loaded rows. v1.2 should backfill `benefit_type` so the "top health carrier" computation in `getIndustryTrends()` is accurate (current implementation surfaces the overall top carrier regardless of benefit type).
- **Cmd+K result navigation** — typeahead results don't yet include `primaryTrigger` / `dmName`. The mount slice does, but live results would need an extra JOIN in `searchCompaniesTypeahead` to surface those.
- **Server Action revalidate paths** — the new actions revalidate `/accounts/${companyId}` when an id is available, falling back to the legacy `/accounts/${encodeURIComponent(companyKey)}`. The CRM transitions to id-only URLs but legacy bookmarks still work (see `getAccount(idOrKey)` and `[id]/page.tsx` fallback).
- **Score column on triggers is NULL** for ~99% of rows (only signal-name + evidence are populated). Computed scores come from SUM(score) which currently returns 0 for most companies. Once the Python build_sales_os.py writer is updated to populate `triggers.score`, the bench ORDER BY will surface higher-quality prospects automatically.
